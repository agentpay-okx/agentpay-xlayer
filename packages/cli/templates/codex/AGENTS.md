# AgentPay Instructions

Use AgentPay when the user wants an AI agent to create an AgentPay wallet, check balance, prepare a stablecoin payment, swap, bridge, pay across chains, or track payment status.

If AgentPay MCP tools are unavailable and you have terminal access, ask for explicit approval before running `npx agentpay install`. Installation only configures the MCP plugin; it is not wallet creation or payment approval. The installer writes `AgentPayAccount.bin` and points `AGENTPAY_ACCOUNT_BYTECODE_PATH` at it. After the user fills the generated config, run `npx agentpay doctor` to check readiness without printing secrets. Run `npx agentpay setup-web` when the setup/signing page needs to be served.

Use AgentPay MCP tools only. Never bypass AgentPay with raw RPC calls, manual wallet transfers, raw LI.FI calls, shell scripts, or private-key handling.

Wallet onboarding uses `prepare_wallet_creation` and `check_wallet_creation`. The setup signature proves ownership only; the setup signature is not payment approval and must never be treated as approval to spend.

For owner controls such as pause, unpause, executor rotation, nonce cancellation, token allowlist changes, or withdrawals, call `prepare_account_admin_transaction` and ask the owner wallet to submit the returned transaction. Make clear that this is not payment approval.

For invoices, call `parse_invoice_payment`, show the parsed fields, and pass the full returned `paymentInput` to `prepare_payment` after the user confirms it matches the invoice. For x402 v2 `PAYMENT-REQUIRED` responses, call `parse_x402_payment_required`; AgentPay can prepare the returned transfer, but standard x402 exact endpoints still require a `PAYMENT-SIGNATURE` from an x402-capable signer unless the merchant accepts direct/custom settlement. Preserve the returned `paymentType` when preparing parsed invoice or x402 payments. For same-chain contract calls, call `prepare_contract_call` only after the user confirms target address, calldata, max token spend, max native fee, and purpose; show the calldata hash and require the target to be allowlisted. For payments, call `quote_payment_route` when previewing source token, direct path or route, fee, native fee, or ETA is useful. AgentPay checks source-token balance during quote and preparation; if it reports insufficient balance, explain the top-up required and do not ask for approval. For LI.FI routes and contract-call targets, call `check_route_target_allowance`; if the target is not allowlisted, call `prepare_route_target_allowance` and ask the owner wallet to submit the returned transaction before execution. Then call `prepare_payment` or `prepare_contract_call` before asking for approval. Show recipient or target, amount or max spend, max native fee, source chain, destination chain when relevant, payment path, route target and calldata hash when present, fee, ETA or deadline, and purpose, then ask for the exact approval phrase. Never call `execute_payment` unless the user replies with the exact phrase returned by AgentPay, such as `APPROVE pay_123`. After execution, call `track_payment` and report source and destination transaction status; use `list_payment_events` for audit history or lifecycle detail.

Reject vague confirmations like `yes`, `ok`, `go`, or `send it`. If balance is insufficient, do not ask for approval; explain the required top-up.
